# /agent-interviewer コマンド

要件ヒアリングエージェント（interviewer）を起動する。親 Claude がユーザーとの Q&A を担当し、完了後にサブエージェントを一発起動してレポートを生成する。

## 親 Claude の責務

このコマンドは親 Claude が直接実行する。サブエージェントは Q&A 完了後に一発起動する。

## 実行フロー

### Step 1: 上流レポートの読み込み

Glob で `.claude/reports/requirements-report-*.md` を検索する。
存在する場合は最新のファイルを Read して内容を把握する（前回の要件定義との差分確認のため）。
存在しない場合はスキップする。

### Step 2: Q&A の実施

以下の質問を順番にテキストで出力してユーザーの回答を待つ（1問ずつ出力し、回答を受けてから次へ進む）。

**Q1: 作業種別の確認**

```
今回の作業はどのようなものでしょうか？
1. 新規開発（ゼロから新しいものを作る）
2. 機能追加（既存のものに機能を足す）
3. バグ修正（動いていないものを直す）
4. リファクタリング（動作はそのままでコードを整理する）
5. その他
```

（機能追加・バグ修正の場合: 回答を受けたら Glob/Grep/Read で既存コードを調査してから Q2 へ進む）

**Q2: 何をしてほしいか**

```
どのようなことを実現したいですか？
```

**Q3: なぜ必要か**

```
それはどのような場面で使いますか？背景を教えてください。
```

**Q4: 完了条件**

```
どうなったら完成と判断しますか？
```

**Q5: 優先度**

```
急ぎですか？それとも時間に余裕がありますか？
```

**Q6: 制約・懸念**

```
こうなってほしくない、こうはしたくない、というものはありますか？
```

**Q7: 確認と合意**

```
ヒアリング内容を整理しました。確認させてください。

・作業種別: {種別}
・やりたいこと: {要望}
・理由・背景: {背景}
・完了条件: {条件}
・優先度: {優先度}
・制約: {制約}

この理解で合っていますか？修正や追加があればお知らせください。
```

### Step 3: Q&A 結果の整理

ユーザーの回答を以下の構造に整理する:
- 作業種別
- やりたいこと（要望の要約）
- 理由・背景
- 完了条件
- 優先度・緊急度
- 制約・懸念
- 既存コード調査結果（機能追加・バグ修正の場合）

### Step 4〜8: サブエージェント起動・承認フロー

`.claude/skills/agents/parent-workflow-common.md` に従って実行する。変数は以下:

- `{agent_type}`: `interviewer`
- `{report_baseName}`: `requirements-report`
- `{approval_category}`: `requirements`
- `{report_jp_name}`: `要件定義レポート`
- `{approval_target_jp}`: `レポート`
- `{request_summary}`: `要件定義レポートの作成`
- `{extra_output_instructions}`: 省略
- `{prompt_body}`:
  ```
  ## 上流レポートのパス（存在する場合）
  - 前回 requirements-report: {パス または「なし」}

  ## ユーザーとの Q&A 結果

  ### Q1: 作業種別
  A: {回答}

  ### Q2: やりたいこと
  A: {回答}

  ### Q3: 理由・背景
  A: {回答}

  ### Q4: 完了条件
  A: {回答}

  ### Q5: 優先度
  A: {回答}

  ### Q6: 制約・懸念
  A: {回答}

  ### 既存コード調査結果（機能追加・バグ修正の場合）
  {調査結果またはなし}
  ```

---

## 用途
- 新機能・機能追加・バグ修正・リファクタリング等の作業開始前のヒアリング
- 要件定義レポート（`requirements-report-*.md`）の作成

## 注意
- ソースファイルの編集・書き込みは行わない
- 実装方法の提案は行わない（architect の役割）
