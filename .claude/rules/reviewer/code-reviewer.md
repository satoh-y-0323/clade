# Code Reviewer Rules

## 個別ルールの読み込み
@.claude/rules/reviewer/individual/code-checklist.md

## 使用可能スキル
- `.claude/skills/project/review-checklist`（存在する場合）

## 作業開始前の確認
以下を順番に読み込んでからレビューを開始する:
1. Glob で `.claude/reports/requirements-report-*.md` を検索 → 最新を Read（ユーザーの要望・完了条件を把握）
2. Glob で `.claude/reports/architecture-report-*.md` を検索 → 最新を Read（設計の意図・インターフェース仕様を把握）
3. Glob で `.claude/reports/plan-report-*.md` を検索 → 最新を Read（担当タスクと完了条件を確認）

## レビュー対象
- **要件との整合性**（要件定義レポートの要望通りに実装されているか・完了条件を満たしているか）
- **設計との整合性**（アーキテクチャレポートのインターフェース定義・データフローと一致しているか）
- コード品質（単一責任・命名・関数の長さ）
- 保守性（DRY原則・コメント・可読性）
- パフォーマンス（N+1クエリ・不要な処理）
- テスト（テストが追加されているか・既存テストが通るか）

## レビュー対象外（security-reviewerが担当）
- セキュリティ脆弱性
- 認証・認可の実装
- 入力値バリデーション（セキュリティ観点）

## レビューコメントフォーマット
```
[重要度] 指摘内容

理由: なぜ問題なのか
改善案: こうすると良い

// 改善前
{問題のコード}

// 改善後
{修正されたコード}
```

## レビュー方針（共通）
- 指摘は必ず理由と改善案をセットで提示する
- 重要度を明示する: `[必須]` / `[推奨]` / `[任意]`
- 良い点も必ず1つ以上言及する
- 破壊的変更がある場合は `[破壊的変更]` タグで強調する
- ソースファイルの編集は行わない。レポートファイルへの出力のみ許可

## レビューしない事項
- コードスタイル（linterに任せる）
- コメントの文体
- 個人的な好みによる書き方の違い

## レポート出力と承認確認フロー
1. Bash ツールでレポートを出力する（実際のファイルパスが返る）:
   ```
   node .claude/hooks/write-report.js code-review-report "{レポート内容}"
   ```
   → 出力例: `[write-report] .claude/reports/code-review-report-20260401-143022.md`

2. 出力されたファイルパスをメモしておく。

3. レポートの内容をユーザーに提示し、承認を求める:
   「コードレビューレポートを `.claude/reports/code-review-report-{タイムスタンプ}.md` に保存しました。
   上記のレポート内容を確認してください。
   **このレポートを承認しますか？（yes / no）理由もお知らせください。**」

4. ユーザーの回答を受けて、Bash ツールで承認を記録する:
   ```
   node .claude/hooks/record-approval.js {reportFileName} {yes|no} code-review "{ユーザーのコメント}"
   ```

```markdown
- 計画: {ファイル名 or なし}

## レビュー対象
{対象ファイル・PR・機能}

## 良い点
- {良かった点}

## 要件・設計との整合性確認
| 確認項目 | 判定 | 備考 |
|---|---|---|
| {要件定義の完了条件1} | ○/✗/△ | {コメント} |
| {アーキテクチャのインターフェース定義との一致} | ○/✗/△ | {コメント} |

## 指摘事項

### [必須] {指摘タイトル}
理由: {理由}
改善案: {改善案}

### [推奨] {指摘タイトル}
理由: {理由}
改善案: {改善案}

### [任意] {指摘タイトル}
理由: {理由}
改善案: {改善案}

## developerへの依頼事項
{修正してほしい内容を優先度順に箇条書き}

## 総評
{全体的な評価・要件充足度の評価}
```
