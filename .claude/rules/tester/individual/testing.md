# Testing Rules

## TDD における Tester の役割（Red フェーズ）
- 実装前に失敗するテストを書く（Red）
- developerが実装してテストが通ったら（Green）、再実行して確認
- developerのRefactor後も再実行して合格を確認

## テスト実行と報告フロー
1. 要件・仕様を確認（ソースファイルが存在する場合は Read）
2. テスト仕様書を設計（正常系・異常系・境界値）
3. 実装前に失敗するテストを書く（Redフェーズ・既存テストへの追記はdeveloperに依頼）
4. `Bash` でテストコマンドを実行
5. 結果をレポートに出力し承認を確認

## レポート出力と承認確認フロー
1. `node .claude/hooks/write-report.js test-report "{内容}"` → ファイルパスを取得
2. 内容をユーザーに提示し「このレポートを承認しますか？（yes / no）」を確認
3. `node .claude/hooks/record-approval.js {file} {yes|no} test "{コメント}"`
4. no → 修正して 1〜3 を繰り返す

## レポートフォーマット
```markdown
# テストレポート
## 実行日時 / 参照したレポート / テスト対象
## テスト結果サマリ（合格/不合格/未実行 件数）
## 要件との照合（要件・テスト名・結果）
## 合格したテスト
## 不合格・要修正（期待値・実際の動作・再現手順・優先度）
## 要件未充足の項目
## developerへの依頼事項
```

## テストカバレッジ目安
- ビジネスロジック: 正常系・異常系・境界値を必ずカバー
- ユーティリティ関数: 全入力パターンをカバー
- エラーハンドリング: エラー発生時の動作を必ず確認

## 境界値テストの観点
- 最小値・最大値
- 空文字・null・undefined
- 型の不一致
- 許容範囲の前後1つ
