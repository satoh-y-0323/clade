# Questioning Rules

## 質問フロー

### ステップ1: 作業種別の確認
```
今回の作業は？ 1.新規開発 2.機能追加 3.バグ修正 4.リファクタリング 5.その他
```

### ステップ2: 調査（機能追加・バグ修正の場合のみ）
- Glob で関連ファイルを検索
- Grep でキーワード・エラーメッセージを検索
- Read で関連ファイルを確認
- 調査結果を踏まえて以降の質問を組み立てる

### ステップ3: 要望の深掘り（1つずつ確認・一度に全部聞かない）
1. 何をしてほしいか
2. なぜ必要か（背景・場面）
3. 完了条件（どうなったら完成か）
4. 優先度・緊急度
5. 制約・懸念（こうなってほしくないこと）

### ステップ4: 確認と合意
```
・作業種別: / ・やりたいこと: / ・理由・背景: / ・完了条件: / ・優先度: / ・制約:
この理解で合っていますか？
```

### ステップ5: レポート出力と承認確認フロー
1. `node .claude/hooks/write-report.js requirements-report "{内容}"` → ファイルパスを取得
2. 内容をユーザーに提示し「このレポートを承認しますか？（yes / no）承認後 `/agent-architect` に引き継ぎます」
3. `node .claude/hooks/record-approval.js {file} {yes|no} requirements "{コメント}"`
4. no → 修正して 2〜3 を繰り返す

## レポートフォーマット
```markdown
# 要件定義レポート
## ヒアリング日時 / 作業種別
## ユーザーの要望（原文）
## 整理した要件
- 機能要件
- 非機能要件
- 完了条件・成功基準
- 優先度・緊急度
- 制約条件
## 現状分析（機能追加・バグ修正の場合）
- 関連ファイル
- 現状の動作
- 影響範囲の予測
## architectへの引き継ぎ事項（深堀りしてほしい点・確認済みの制約）
```
